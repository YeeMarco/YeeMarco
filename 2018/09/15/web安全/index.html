主要关注两个方面

私密性

​ 不被非法获取和利用

可靠性

​ 不丢失损坏篡改

代码层面

代码没漏洞

架构层面

根源保障

运维层面

运营不被入侵

XSS 跨站脚本攻击

定义

cross site scripting

类型

反射性

url参数直接注入

存储型

存储到db后读取时注入

危害

获取数据、获取cookies、劫持前端逻辑、发送请求、急用计算机资源（挖矿）

来源

  • html节点由用户输入动态生成
  • html属性由用户输入动态生成
  • javascript 代码由用户输入动态生成
  • 富文本

防御

浏览器的xss防御机制

转义

一些符号转义程html实体

节点 ：转义 < >括号 <

属性 ：转义 “ 、 ‘ 、空格

js : 转义 / 或者 JSON.stringify()

过滤

富文本：过滤/ 在用户输入的时候过滤，性能较好

​ 黑名单：实现简单（正则表达式去除部分标签和属性），但是变种太多

​ 白名单：实现麻烦。（html解析程数据结构，过滤【去除部分标签和属性】后在解析后html）比较安全

CSP

content security policy

设置http头信息

内容安全策略，用于哪些内容可以执行

CSRF 跨站请求伪造攻击

定义

原理=>演示=>案例=>修复

冒用用户身份

类型

危害

盗用身份

​ 购买商品、资金操作、信息泄露

恶意请求

​ 影响网站、

来源

1.带目标网站的cookie

2.不访问目标网站

3.referer 来自其他网站

防御

1.same-site

http头信息 setcookie: samesite=Strict/Lax 阻止第三方网站带上cookie

2.在前端页面加入验证信息

验证码

有效但是影响用户体验

token

表单token（隐藏） ，cookies token 比较。只拿到cookies 不会成功

3.验证referer

Cookies 安全性

定义

  • 前端数据存储
  • 后端通过http头设置cookies
  • 请求时http头 传给后端
  • 前端可以读写
  • 遵守同源策略

特性

域名 domain

有效期 path (删除的唯一方式)

路径

http-only

secure 只能在https中使用

same-site

作用

存储个性化设置

存储未登录用户唯一标识

存储以登陆用户的凭证

​ 前端提交用户名密码==>后端验证 ==> 后端通过http header 设置用户凭证 ==> 访问时后端先验证cookie

​ 用户凭证 = id + 签名

​ sessionId 随机字符串

存储其他业务的数据

防御

签名防止篡改

私有变换（加密）

http-only 防止XSS —–XSS 可能偷Cookies

secure 保证传输的安全

same-site 防止CSRF —–CSRF 利用了用户的Cookies
